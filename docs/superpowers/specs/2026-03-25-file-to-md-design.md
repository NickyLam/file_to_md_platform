# 文件转 Markdown 平台设计

## 1. 背景与目标

本项目要提供一个浏览器可用的文件转 Markdown 服务，首版聚焦于高质量结构还原，而不是只做“能转出来”。

核心目标：

- 支持通过网页上传文件并获得 Markdown 结果
- 首版支持 `docx / pdf / xlsx`
- 尽量保留原文档结构，包括标题、列表、表格、图片引用等
- 对 `PDF` 的扫描件提供可选 OCR 能力
- OCR 后处理支持可选大模型增强，但不是运行硬依赖
- 不做登录
- 记录审计信息，但不记录正文内容
- 架构上保持可扩展，便于后续增加 `pptx`、更多 OCR 引擎、对象存储和任务管理能力

非目标：

- 不在首版支持 `pptx`
- 不做团队空间、共享协作、权限系统
- 不做正文内容级别的审计留存
- 不做强实时、低延迟流式转换

### 1.1 约束与默认值

以下策略首版应作为可配置默认值，而不是写死在代码里：

- 单文件大小上限
- 任务结果与原始文件保留时长
- OCR 是否启用
- 大模型后处理是否启用

OCR 和后处理过程中产生的中间内容只允许短期存在于进程内存或临时文件中，不作为审计正文长期保留。

## 2. 需求范围

### 2.1 输入范围

首版只支持以下文件类型：

- `docx`
- `pdf`
- `xlsx`

### 2.2 输出范围

输出内容包括：

- Markdown 文件
- 在线预览内容
- 转换任务状态
- 错误信息与排障码

### 2.3 质量目标

转换结果优先满足以下要求：

- 保留文档标题层级
- 保留段落与列表结构
- 保留表格内容和基本结构
- 对图片输出引用或占位信息
- 对 PDF 扫描件尽量恢复出可读 Markdown

### 2.4 输出契约

首版 Markdown 输出应按以下规则保持一致：

- 采用 `CommonMark` 语义，并允许使用 `GFM` 表格语法
- 标题尽量按原始层级映射为 `#` 到 `######`
- 列表尽量保持有序与无序层级
- 表格优先输出为 Markdown 表格，无法稳定展开时降级为分块文本
- 图片优先输出为链接或占位文本，不强制嵌入二进制内容
- `xlsx` 多工作表输出时，每个工作表单独生成一个 Markdown 小节
- 合并单元格需要在 Markdown 中保留可读性，必要时用重复值或说明文本降级

## 3. 用户体验

用户通过浏览器完成以下操作：

1. 选择或拖拽文件上传
2. 查看任务是否创建成功
3. 等待后台转换完成
4. 预览 Markdown 结果
5. 下载 Markdown 文件
6. 查看失败原因与基础排障信息

首版不要求登录。浏览器侧会生成一个匿名 `device_id`，用于识别同一设备的后续请求。

### 3.1 访问模型

上传后，API 服务应返回一个与任务绑定的匿名访问凭证，例如 `access_token`。

访问规则：

- 任务状态查询、预览与下载都必须携带 `access_token`
- `task_id` 不能单独作为访问凭证
- `device_id` 只用于审计和弱关联，不作为唯一安全边界
- `access_token` 应与任务和 `device_id` 绑定

这样可以在不做登录的前提下，避免公开可猜测的任务地址直接暴露结果。

## 4. 总体架构

系统采用“前后端分离的单体应用 + 异步任务”的方式实现。

### 4.1 组件划分

- 前端 Web：负责上传、轮询任务状态、预览结果、下载文件
- API 服务：负责接收上传、创建任务、查询任务、提供结果下载和日志写入
- Worker：负责执行文件解析、OCR、Markdown 组装
- Redis：负责任务队列和任务状态缓存
- PostgreSQL：负责任务元数据、审计日志、失败信息持久化
- 文件存储：负责保存原始文件、转换结果和短期中间产物

### 4.2 推荐技术栈

- Python `3.13.12`
- FastAPI
- Redis
- PostgreSQL
- Docker Compose

### 4.3 部署原则

- 首版必须可在多平台通过 Docker 部署
- Docker Compose 作为默认启动方式
- 文件存储优先使用本地卷，后续可替换为 MinIO 或 S3

## 5. 核心流程

### 5.1 上传与任务创建

1. 用户上传文件
2. 前端做基础校验，包括文件类型、文件大小和空文件检测
3. API 服务接收文件并再次校验文件类型、文件大小、内容签名和空文件检测
4. API 服务计算文件哈希
5. API 服务记录任务元数据
6. 原始文件落盘
7. API 服务把转换任务写入 Redis 队列
8. API 服务返回任务 `id` 和 `access_token`

### 5.2 后台转换

1. Worker 消费任务
2. 根据文件类型路由到对应转换器
3. 执行基础结构抽取
4. 必要时进入 OCR 流程
5. OCR 结果可选进入大模型后处理
6. 组装 Markdown
7. 保存结果文件
8. 更新任务状态
9. 记录耗时、失败信息和引擎版本

### 5.3 结果展示

1. 前端轮询任务状态
2. 任务成功后展示 Markdown 预览
3. 用户可下载最终 Markdown 文件
4. 失败时展示错误信息和排障码

### 5.4 任务状态

任务状态建议至少包含：

- `pending`
- `running`
- `success`
- `failed`

必要时可增加：

- `queued`
- `retrying`
- `canceled`
- `success_with_warnings`

首版实现不要求所有状态都必须暴露给用户，但后端状态机应保留扩展能力。

### 5.5 部分成功语义

部分成功适用于基础转换完成，但某些增强步骤失败或降级的情况。

建议判定方式：

- `pdf` 基础转换已经生成至少一个有效 Markdown 块时，哪怕 OCR 或大模型后处理失败，也可返回 `success_with_warnings`
- `pdf` 基础转换没有生成任何有效 Markdown 块，且 OCR 不可用、超时或失败时，应返回 `failed`
- `docx` 和 `xlsx` 的基础转换若已经生成有效 Markdown 块，则增强步骤失败只影响为 `success_with_warnings`
- 若基础转换输出为空、仅有占位符、或无法形成任何有效 Markdown 块，则必须返回 `failed`

`success_with_warnings` 的结果仍应允许用户下载，只是需要展示降级提示。

## 6. 转换器设计

转换逻辑按文件类型拆分，避免把所有解析逻辑集中在一个大模块里。

### 6.1 `docx` 转换器

职责：

- 读取文档段落、标题、列表、表格和图片引用
- 尽量还原章节层级
- 输出结构化 Markdown

关注点：

- 标题层级映射
- 列表嵌套
- 表格转 Markdown 表格或分块文本
- 图片位置和引用信息

### 6.2 `xlsx` 转换器

职责：

- 解析工作表内容
- 识别表头、数据区、合并单元格
- 优先保留适合 Markdown 的表格结构

关注点：

- 多工作表输出策略
- 合并单元格展开方式
- 空行、空列、批注和隐藏行列的处理策略

### 6.3 `pdf` 转换器

职责：

- 优先提取文本型 PDF 的正文
- 在文本抽取不足时进入 OCR
- 对 OCR 结果做结构整理

关注点：

- 文本型 PDF 与扫描件 PDF 的区分
- 分页结构处理
- OCR 可插拔
- 大模型后处理可选

### 6.4 OCR 与大模型后处理

OCR 不是强依赖。系统默认应能在不接大模型的情况下运行。

建议链路：

- OCR 引擎负责字符识别
- 大模型可选用于后处理
- 后处理的职责是修复段落断裂、标题层级和简单表格结构

大模型只作为增强插件，不作为基础运行条件。

## 7. 数据与存储

### 7.1 持久化对象

数据库持久化以下内容：

- 任务记录
- 文件元数据
- 审计日志
- 失败原因码
- 引擎版本信息

### 7.2 文件存储

文件本体不入数据库。

存储内容包括：

- 原始上传文件
- Markdown 结果文件
- 短期中间产物

### 7.3 保留策略

原始文件、结果文件和审计日志的保留时长应通过配置控制。

建议首版默认策略为：

- 原始文件和结果文件可在配置期满后清理
- 审计日志默认保留更长时间
- 临时中间产物在任务结束后尽快清理

具体保留周期不在首版强制固定，由部署方通过环境变量或配置文件决定。

### 7.4 访问范围

存储的原始文件、结果文件和审计日志只允许服务端和授权运维角色访问。

首版不需要前台用户之间的共享访问，也不需要公开可浏览的结果地址。

### 7.5 哈希与关联

每个上传文件都需要计算文件哈希，用于：

- 去重或幂等控制
- 审计追踪
- 失败排查

## 8. 审计与合规

首版需要记录审计信息，但不记录正文内容。

### 8.1 记录内容

- 客户端 `IP`
- 匿名 `device_id`
- 文件名
- 文件大小
- 文件类型
- 文件哈希
- 任务状态
- 任务耗时
- 失败原因码
- OCR 是否启用
- 大模型后处理是否启用
- 转换引擎版本

### 8.2 不记录内容

- 原始正文
- OCR 完整正文
- Markdown 完整正文

### 8.3 `device_id` 定义

`device_id` 不代表真实硬件地址，不要求读取 MAC 地址。

建议实现为：

- 浏览器首次访问时生成匿名标识
- 存储在本地持久化位置
- 发送给后端用于关联同一设备的审计记录

### 8.4 审计日志访问范围

审计日志只允许管理员或运维角色查看。

首版不提供普通用户查询历史审计记录的能力。

## 9. 错误处理

错误分为四类：

- 上传错误
- 解析错误
- 后处理错误
- 系统错误

### 9.1 上传错误

示例：

- 文件类型不支持
- 文件过大
- 空文件
- 文件损坏

处理方式：

- 同步返回
- 前端提示可读错误
- 后端必须再次校验文件类型、大小和文件内容
- 仅靠前端校验不作为可信边界

校验规则：

- 扩展名、客户端声明的 MIME 和服务端嗅探结果不一致时，优先以服务端嗅探结果为准
- 路由转换器时优先使用服务端识别结果
- 不支持的文件一律拒绝

### 9.2 解析错误

示例：

- `docx` 结构异常
- `xlsx` 读取失败
- `pdf` 无法提取文本

处理方式：

- 任务状态标记失败
- 保留失败原因码
- 允许用户重新上传

### 9.3 后处理错误

示例：

- OCR 服务不可用
- 大模型调用失败
- 结构修复结果异常

处理方式：

- 记录到任务失败信息
- 对可重试错误支持重试
- 不影响基础转换链路的可用性

如果任务已经产生可用 Markdown，只是增强步骤失败，则应按 `success_with_warnings` 处理，而不是直接失败。

### 9.4 系统错误

示例：

- 队列不可用
- 数据库写入失败
- 磁盘空间不足

处理方式：

- 返回通用失败
- 记录系统级错误码
- 由运维层告警

## 10. 测试策略

### 10.1 单元测试

覆盖：

- 文件类型识别
- 哈希计算
- 各文件转换器的基础逻辑
- 审计日志字段生成

### 10.2 集成测试

覆盖：

- 上传后任务创建
- 队列消费
- 状态流转
- 结果落盘与读取
- 任务失败路径

### 10.3 样本回归集

需要准备一组固定样本文件：

- 正常 `docx`
- 含多级标题的 `docx`
- 含表格的 `docx`
- 正常 `pdf`
- 扫描件 `pdf`
- 简单 `xlsx`
- 含合并单元格的 `xlsx`

### 10.4 失败测试

覆盖：

- 错误格式文件
- 损坏文件
- 超大文件
- OCR 不可用
- 大模型不可用

### 10.5 验收阈值

首版回归测试至少应检查以下内容：

- `docx` 样本集全部应达到 `success` 或 `success_with_warnings`
- `xlsx` 样本集全部应达到 `success` 或 `success_with_warnings`
- `pdf` 文本型样本全部应达到 `success` 或 `success_with_warnings`
- 扫描件 `pdf` 在 OCR 启用时，至少应有一份样本达到 `success`
- OCR 或大模型不可用时，已产生有效 Markdown 块的任务应进入 `success_with_warnings`，否则应进入 `failed`
- 所有任务的状态查询、预览和下载都必须能通过 `access_token` 正常完成
- 回归集中的样本在输出中应保留可识别的标题层级、表格结构或工作表分隔信息，至少满足人工阅读和定位原内容的需要

## 11. 部署方案

### 11.1 默认部署形态

使用 Docker Compose 部署以下服务：

- Web 前端
- API 服务
- Worker
- Redis
- PostgreSQL

### 11.2 部署目标

- Linux 服务器
- macOS 本机
- Windows 通过 Docker Desktop 或 WSL

### 11.3 环境配置

通过环境变量控制：

- 数据库连接
- Redis 地址
- 存储路径
- OCR 开关
- 大模型开关
- 日志级别

## 12. 后续扩展点

首版不实现，但需要预留的扩展点：

- `pptx` 支持
- 更多 OCR 引擎
- 对象存储替换本地卷
- 用户登录与权限
- 团队空间
- 转换历史管理
- 任务重试与任务队列监控面板

## 13. 结论

本方案建议首版采用：

- `Python 3.13.12`
- `FastAPI`
- `Redis`
- `PostgreSQL`
- `Docker Compose`
- 异步转换任务
- 模块化文件转换器

这能在控制首版复杂度的前提下，满足结构化 Markdown 转换、审计记录和未来扩展需求。
